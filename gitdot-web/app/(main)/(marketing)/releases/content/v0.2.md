There are two things we know we must do rather quickly:

**1. Infrastructure**
As of writing, gitdot uses NFS.

We chose to do so for faster iteration time, as we did not want to manage persistent disks, which would require sticky routing and a rather complicated infastructure redesign.
NFS does not play well with git, which assumes that its underlying file system is very fast. This is most obvious when doing things like running `git diff` which executes many recursive file stats and reads. 

So **TL;DR:** git push is slow, we'll make it faster.


**2. Issues**
There is not a way to report issues besides emailing us directly.
That is pretty bad, and a tad comical.

We'll push out a barebones version of a bug tracker prior to the launch (likely mid June or so), but will take our time to iterate and properly a design a bug tracking system centered around a few core use-cases. We don't want to re-invent the wheel here, but neither do we think we should assume that project management paradigm is what OSS projects need.
